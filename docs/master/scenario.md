# バビルサ研究所ゲームパート概説(シナリオ製作編) {#title}

2025/10/19

この章では、シナリオ製作にとりかかるにあたって、推奨される……というより、筆者がやってほしいシナリオの書き方について記載する。

筆者はシナリオを書き、みんなと共有するにあたり、重要な要素は以下の3点であると考える。

- ファイル形式の統一
- 書き方の統一
- 共有方法の統一

これらを統一することで、シナリオをスムーズに、かつ可読性高く共有することが出来、快適なセッションにつなげることができる。
とはいえ、慣れないファイルの書き方では苦戦する場面も多々あるだろう。
以下に望ましい書き方とそのコツを記載するので、参考にしてほしい。

## ファイル形式 {#file_format}

共有するシナリオのファイル形式は「**Markdown**(マークダウン)」にしてほしい。

### Markdownとは? {#what_is_markdown}

Markdownとは、簡略化・形式化されたテキストファイルの書き方、及びその書き方で記載されたテキストファイルの形式である。
Markdownを利用すると、統一された書式でシナリオを作成でき、Officeファイルを使うことによるめんどくさい点やリスクもある程度回避することができる。
また、PDFやHTMLに変換することが簡単というメリットもあるので、作ったシナリオを印刷して持ち歩きたい時にもとても便利なのだ。
~~そんな機会が本当にあるかは一旦置いておいて。~~

ちなみに、Discordのテキストメッセージで効果を加えるときも、Markdown形式が使われている。
もしテキストメッセージの効果を普段から多用している場合は、想像よりはすんなり書けるようになるかも知れない。
もちろん、そうじゃない方向けに、シナリオ作成時に必要な書式は「書き方」の章で明記しているので、安心して欲しい。

```markdown
<!-- Markdown形式のファイル名は、こんな感じ。 -->
(任意のファイル名).md
```

## 書き方 {#how_to_write_scenario}

### 基本編 {#basic}

Markdownの記法では、各種見出しの前に`#`(シャープ)マークをつけることで、章の位置を区別している。
また、先頭につけるシャープの数を、直前のシャープの数より一つ増やすと、一つ前の章の小見出しとして表現することも出来る。
もちろん小見出しの小見出しを作ることも可能であり、最大6回まで小見出しを表現出来る。

こと本作のシナリオにおいては、見出し機能を使って、「タイトル」「章区切り」「イベント」「各種データ」の見出しを作成する。
具体的には、このような書き方をしてほしい。

```markdown
# (シナリオタイトル)

## (章番号): (章タイトル)

### (データ名)

#### (データ詳細)

```

各種用語の意味は以下の通りである。

|用語名|意味|
|:--|:--|
|シナリオタイトル|シナリオのタイトル。シナリオのテーマや要約、伏線などを1行の文章で表そう。|
|章番号|各種章につけられる連番。セーブポイントの区切りにもなる。短期シナリオの場合は第1章だけでも書くこと。書き方はお好みで問題ないが、「第X章」という形式がオススメ。。|
|章タイトル|その章のタイトル。印象に残したい場面を1行で要約しよう。|
|データ名|イベントの名前、NPCの名前、環境設定、戦闘で手に入るアイテム表、買物イベントのラインナップ、Etc.とにかくいろいろなデータはシャープ4つで表す。具体的な書き方は都度説明するので、安心して欲しい。|
|データ詳細|データ名に紐付く詳細データ。書き方は内容によって変わる。|

### 探索パート編 {search_part}

探索パートでは、調べた場所に応じて発生するイベントを、表形式で表す必要がある。
そのため、テーブル構文を利用する。

テーブル構文は、表を表すための構文である。
少し複雑であるが、細かく切り分けて説明する

```markdown
|商品|在庫|単価|
|:--|:--|:--|
|りんご|3|200円|
|いちご|2|300円|
|バナナ|5|150円|
```

共通
:   行方向のデータは改行で区切り、列方向の区切りは`|`(パイプライン)という記号で区切る。
    円マークを改行すると出てくるので、覚えておくと良いだろう。

1行目(ヘッダー)
:   ヘッダーにはデータのタイトルを記載する。
    タイトルは各種書き方に用意されているので、それをコピー&ペーストすればよい。

2行目
:   データが右揃えか左揃えかなどを設定する行。
    例の書き方の場合は左揃えになる。

3行目以降
:   実際のデータを表す。
    基本的には文章や文字列をそのまま書けばよい
    特殊な記法は、出てくるごとに解説する。

#### マップ探索 {#map_search}

設定項目はおもに2種類。

- 描写表
- イベント表

```markdown
### 描写表：(場面名)

|調べる場所|調べる座標|描写|
|:--|:--|:--|
|机|(3,24)|埃をかぶっている。インクが乾いてしまっているものの、掃除すればまだまだ使えそう。開けられる引き出しにはなにもないが、カギのかかった引き出しがある。|
|本だな|(12,4)|机と比べると手入れされている。中の本は技術書だが、どれもこれも情報が古すぎて使い物にならない。叩くと裏に空洞があるような音がする。|

### イベント表：(場面名)

|調べる場所|調べる座標|発生イベント|発生方式|必要アイテム|必要スペシャルムーブ|必要フラグ|
|:--|:--|:--|:--|:--|:--|:--|
|机|(3,24)|薬品発見|近くで調べる|机のカギ|||
|本だな|(12,4)|隠し扉発見|スキルを使う||ファイア・ボール||
|(任意)|(任意)|セルリアン乱入|3回連続で調査を行う||||
```

#### イベント表の説明 {#event_table_detail}

場面名
:    探索パートが発生する場面。

調べる場所
:    プレイヤーに調べられるのを想定している場所。

調べる座標
:   実際に調べられる座標。
    移動中に発生するトラップ以外は、厳密に遵守しなくてもよい。

描写
:   調べたときに開示するべき情報。
    要点さえあっていれば、多少アレンジしてもよい。

発生イベント
:   その場所を調べたときに発生するイベント。
    イベントはこの表の近くで別途定義する。

発生方式
:   どのような形でそのイベントが発生するかを定義する。
    同じ場所の調査でも、調べ方によって発動するイベントを変えることができる。
    主に使うモノ
    - 見る
    - 近くで調べる
    - スペシャルムーブを使う
    - アイテムを使う
    - 自動発動

必要アイテム・必要スペシャルムーブ
:   イベントを発生させるために必要なスキル。
    いずれも類似する効果を持つもので代替も可能。柔軟に対応しよう。

必要フラグ
:   事前に発動させるべきイベント。

#### テキスト探索 {#text_search}

```markdown
### 描写表：(場面名)

|調べる場所|描写|
|:--|:--|
|机|埃をかぶっている。インクが乾いてしまっているものの、掃除すればまだまだ使えそう。開けられる引き出しにはなにもないが、カギのかかった引き出しがある。|
|本だな|机と比べると手入れされている。中の本は技術書だが、どれもこれも情報が古すぎて使い物にならない。叩くと裏に空洞があるような音がする。|

### イベント表：(場面名)

|調べる場所|発生イベント|発生方式|必要アイテム|必要スペシャルムーブ|必要フラグ|
|:--|:--|:--|:--|:--|:--|
|机|薬品発見|近くで調べる|机のカギ|||

```

基本的にはマップ探索と同一だが、注意するべき点が2点ほどある。

調べる座標
:   テキスト探索では文章のみで話が展開していくので、記載の必要はない。

発生方式
:   特定の座標に来たときに自動発動する発生方式は利用できない。

### 戦闘パート編 {#battle_part}

戦闘パートでは、リスト構文とテーブル構文を利用する。

リスト構文とは、簡単に言うと箇条書きを表すための構文であり、行頭に`-`を書くことで表現出来る。
ただし、

- 同じデータを表すリスト同士には改行を含んではいけない
- リストは空白行で隔離する必要がある

などの制約もある。

```markdown
- りんご
- いちご
- バナナ
```

#### 敵のデータ(必須データのみの場合) {#enemy_data}

敵のデータには、以下の4点を表す必要がある。

- ステータス
- パラメータ
- 行動パターン
- スキル

```markdown

### 敵：(敵の名前)

#### (敵の名前)：ステータス

- HP: (最大値)
- SS: (初期値)
- 人間度: (初期値)
- 移動速度: (初期値)

#### (敵の名前)：パラメータ

- 筋力: (値){修正値}
    - 格闘: (値)
    - 投擲: (値)
- 敏捷力: (値){修正値}
    - 射撃: (値)
- 魔力: (値){修正値}
    - 魔法: (値)
- 判断力: (値){修正値}
    - 回避: (値)
    - 精神抵抗: (値)
- 耐久力: (値){修正値}
    - 肉体抵抗: (値)
- 生態情報料: (値)
- 経験値: (値)

#### (敵の名前)：行動パターン

1. (行動1)
    - (行動1の説明文)
1. (行動2)
    - (行動2の説明文)

#### (敵の名前)：スキル

|行動|効果|描写|発動条件|
|:--|:--|:--|:--|
|(行動)|(効果)|(描写)|(発動条件)|
|(行動)|(効果)|(描写)|(発動条件)|
:

```

#### スキルの表の説明 {#skill_table_detail}

行動
:   スキルの名前

効果
:   このスキルが戦況に与える影響。
    ダメージや射程距離、与えるバフデバフ状態異常……などは、すべてここに書く。

描写
:   スキルのセッション上における描写。
    その敵がどのようにして、戦況に効果を与えるか。

発動条件
:   発動に必要な条件はあるか。
    ない場合は何も書かなくてもよい。

```markdown

### 敵グループ：(敵グループの名前)

#### (敵グループの名前)：メンバー

- (敵の名前1)
- (敵の名前2)
:

#### (敵グループの名前)：ドロップアイテム表

|落とす敵|アイテム|落とす出目|
|:-|:-|:-|
|(敵の名前1)|(ドロップアイテム1-1)|(出目)|
|^|(ドロップアイテム1-2)|(出目)|
|^|はずれ|(出目)|
|(敵の名前2)|(ドロップアイテム2-1)|(出目)|
|^|(ドロップアイテム2-2)|(出目)|
|^|はずれ|(出目)|

```

##### ドロップアイテム表の説明 {#drop_item_table_detail}

- 敵名
    - ドロップアイテムに対応する敵の名前。
    - 上の列と同じ敵に設定したい場合は、`^`と書く。
- アイテム名
    - 落とすアイテムの名前。
    - ドロップアイテムなしの場合は`はずれ`と書く。
- 落とす出目
    - 落とす出目。
    - ドロップアイテムは、戦闘終了後に1d100の出目を振り、その値がどのアイテムの範囲内にあったかで決める。
    - 厳密には出目の範囲を定める必要があり、範囲の最小値と最大値を記載する必要がある。
    - `1-10`, `11~30`などと記載するとわかりやすい。

### イベント編 {#event}

イベントは、下記の3点を組み合わせて記載する。

- 条件
- 演出
- 処理

記載にあたってのルールとしては、以下の2点がある。

1. 条件は一番最初に1回のみ記載する。
1. 処理中に演出を挟みたい場合は、『「演出：●●」へ飛ぶ』などの記載を挟まなければならない。逆もまた然り。

#### イベントの例 {#event_example}

```markdown
### イベント：人質救助

#### 人質救助：条件

(条件の記述)

#### 演出A

(演出Aの記述)

#### 処理A

(処理Aの記述)
(演出Bまたは演出Cへ繊維)

#### 演出B

(演出Bの記述)

#### 演出C

(演出Cの記述)
```

#### 条件 {#branch}

条件には、リスト構文を使用する。

##### 条件の例 {#branch_example}

```markdown
#### 図書館突入：条件

- アイテム「図書館のカギ」を持っている

または

- 引き連れたNPCに「ワシミミズク」がいる
```

#### 演出 {#direction}

演出では、引用構文を利用する。
引用構文は、行頭に`>`マークをつけるだけでよい。

それよりも注意して欲しいのは、発言や描写の書き方である。

1. 発言者名と発言、行動内容は`：`(コロン)で挟む。
1. 描写はPM発言として記載する。
1. 発言者名は、`**`(アスタリスク2つ)でかこむ。
    - こうすることで、発言者が強調される。
1. PM描写をする必要がない簡単な行動は`()`で挟む。
1. 発言者が切り替わるごとに、ひとつの改行を挟む。
    - この時、挟まれた改行にも`>`マークをつけるのを忘れずに。

##### 演出の例 {#direction_example}

```markdown
#### 演出：●●村到着

> **PM**：君たちが目的地の村……そう呼んで良いかは定かではないが、集落にたどり着いたら、ある少女が君たちに話かけてきた。
>
> **???**：●●村へようこそ！何もない所だけどゆっくりしてってね！
>
> **PM**：その少女は、少し黄色みがかかった、白金のような毛皮をゆらし、キツネのような耳を跳ねさせながら、ハキハキとこの村の紹介をしてくれた。
> 歓迎されていることは間違いないらしいが、目の隈を見ると、体力的な疲労を隠そうとしていることもわかってしまう。
>
> **ポリーナ**：私は**コサックギツネ**。まわりからは**ポリーナ**って呼ばれてるの。
> この村は平和だよ！**ここのところ村中の人々が嫌な夢ばかり見てたり**とかはしないから、安心してね！
> (大きなあくびをする)
> はっ！ごめんごめん、つい……。
>
> **PM**：……なるほど、疲労の理由は明白である。
> さて、君たちはポリーナにさらに話しかけることもできるし、村でいろいろな人物に話しかけ、装備を調えてもよい。
> 一旦自由行動だ。どうする?
```

連番付きリストを利用するには、行頭に`1.`を記入する。

```markdown
#### 処理：アイテム変更

    - YES

    - NO
        1. 「演出：古びたコインの要求」に移行する
```

## おまけB：なぜOfficeファイルじゃないのか {#appendix_b}

君は、Markdownの説明を聞いて、「別にそんな特殊なファイル使わなくたって、WordとかExcelで良すぎるだろ」と思ったかもしれない。
しかし、WordやExcelなどのOfficeファイルには、下に記したようにいろいろなリスクやめんどくさい要素が存在する。
Markdownはそれらのリスクをまるっと解決してくれる救世主のような存在なのだ。

### Microsoftアカウントの関連情報がもろに残る {#appendix_b_1}

Officeファイルには、内部情報として作成者と編集者のMicrosoftアカウントが自動で登録される。
削除することはもちろん可能だが、うっかり消し忘れて共有すると軽い情報漏洩となる。
特に、アカウント名に本名が設定されていたら、もう目も当てられないことになる。
他にも個人情報が混ざるリスクはけっこう多く存在しており、共有の度にしっかり確認することが求められるのだ。

その点テキストファイルであるMarkdownには、わざと書き記さない限り個人情報が混ざるリスクは低い。
シナリオを共有する際には、これで行った方が気楽には違いないだろう。
共有方法の統一という点において、これは多大なメリットである。

### 書式に関する細かな問題がある {#appendix_b_2}

Wordは、自由に書式を設定して、流麗な文書をお手軽に作成できることがメリットだ。
だが、これは我が研究所の場合、シナリオ制作者が各々自由に書式を決めて、シナリオによって可読性にばらつきが出るリスクに繋がる。
だからといって無理に統一しようとすると、今度はシナリオ制作者に書式設定といういらない負担をかけてしまう可能性がある。
実際に操作するとわかるが、意味のある設定を書式設定で分離するのは、意外と骨が折れる作業なのだ。
以上より、Wordを利用してしっかりしたシナリオを書き記す場合は、細かな点で面倒が生じてしまう。

また、なんならExcelでシナリオを記載することもできる。
だが、Excelはそもそも文書を作るためのツールではなく、読みづらい書式になってしまうリスクも往々にして高い。
ファイル形式が統一されないことも、面倒な要素の一つだ。

他方、Markdownで記載されたシナリオは「Markdownで書く」時点ですでに書式の統一が保証されている。
書式の設定方法についても、頭に記号をつけたり、特殊な記号で囲んだりするだけなので、少なくともWordのように、状況に応じて調整しながらボタンをポチポチするよりは負担は軽いだろう。
これらを考えると、書式の統一においても、Markdownを利用するメリットは非常に大きいと言えるだろう。

### そもそも、Microsoft 365に加入していないと、代替の確認手段が必要になる {#appendix_b_3}

最後に、コレが一番致命的なのだが、そもそもMicrosoft Officeは有料サブスクである「Microsoft 365」というサービスの一部である。
無料で各種資料を使いたいなら、LibreOfficeなどと言った専用の代替ツールをインストールするなり、Web版を使用するなりしなければならない。

その点Markdownであれば、最悪メモ帳でもリッチなテキストが書ける。
共有のハードルは一気に下がると言えるだろう。
